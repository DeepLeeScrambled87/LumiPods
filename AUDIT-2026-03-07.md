# LumiPods Production Readiness Audit

Date: 2026-03-07

## Scope

Review of frontend, backend integration paths, PocketBase schema alignment, content coverage, rewards/gamification, neurodivergent support, and external learning-platform connectors.

## Constraints

- Frontend builds and runs locally.
- `npm run build` passed.
- `npm run lint` failed with existing issues.
- No automated tests were found.
- PocketBase was not running locally during this review.
- Docker Compose exists, but Docker daemon was not running, so live database contents could not be inspected.
- Database findings below are based on schema and service code, not live records.

## Highest-Risk Findings

### 1. Authentication is demo-grade, not production-grade

- Auth is stored in localStorage and can be recreated client-side.
- Learner-to-parent elevation is allowed without credentials.
- Parent login creates a family locally but logs in with a synthetic family ID instead of the created family ID.

Impact:

- Unsafe for real families or learner accounts.
- Breaks data integrity between auth state and stored family state.

Primary files:

- `src/features/auth/AuthContext.tsx`
- `src/features/auth/LoginPage.tsx`
- `pb_migrations/schema.json`

### 2. Family persistence path is broken for new families

- `FamilyContext` creates a new family object and then calls `familyDataService.save()`.
- `familyDataService.save()` always attempts a PocketBase update, not create/upsert.
- When PocketBase is online, newly created families will try to update a record that does not exist.

Impact:

- New signups can silently fail to persist.
- Sync queue can accumulate invalid updates.

Primary files:

- `src/features/family/FamilyContext.tsx`
- `src/services/dataService.ts`

### 3. PocketBase schema and app services are out of sync

- The `progress` schema stores daily aggregates, while `src/services/pocketbase/progressService.ts` expects per-block progress fields such as `blockId`, `dayOfWeek`, `status`, `startedAt`, `focusScore`, and `artifactId`.
- `achievementService` reads and writes an `achievements` collection that does not exist in the schema.
- `portfolioService` uses `learnerId`, `url`, and `artifact_files`; the schema uses `learner`, `externalUrl`, and has no `artifact_files` collection.
- `dataService` artifact mapping omits required `family` and writes `url` rather than `externalUrl`.
- A second artifact service under `src/services/pocketbase/artifactService.ts` is closer to the schema, which means there are competing service layers.

Impact:

- Live PocketBase writes and reads will be inconsistent or fail.
- Portfolio, achievements, and progress tracking are not trustworthy in production.

Primary files:

- `pb_migrations/schema.json`
- `src/services/pocketbase/progressService.ts`
- `src/services/achievementService.ts`
- `src/services/portfolioService.ts`
- `src/services/dataService.ts`
- `src/services/pocketbase/artifactService.ts`

### 4. Many visible user features are still mock-driven

- Dashboard schedule and materials are hardcoded.
- Reports generate fixed sample statistics.
- Student profile relies on mock portfolio, project, achievement, and skills data.
- Portfolio boots from mock artifacts and competencies and only replaces them if live data exists.
- Calendar uses generated example events rather than persisted schedule history.

Impact:

- The product appears more complete than the real data layer supports.
- Families may assume progress is being tracked when it is not.

Primary files:

- `src/features/dashboard/DashboardPage.tsx`
- `src/features/reports/ReportsPage.tsx`
- `src/features/students/StudentProfilePage.tsx`
- `src/features/portfolio/PortfolioPage.tsx`
- `src/features/calendar/CalendarPage.tsx`

### 5. External learning-tool integration is curated-link only

- Partner platforms are modeled as outbound links with an `integrationLevel` field, but current entries are effectively curated links.
- UI renders cards and outbound actions only.
- `trackingEnabled` exists in cross-subject partner session data, but I did not find OAuth, token storage, account linking, job sync, or imported progress logic.

Impact:

- The product cannot yet aggregate real work completed in tools like Khan Academy, Duolingo, Scratch, or Synthesis.
- "Bring learning back into one place" is currently a product promise, not an implemented system.

Primary files:

- `src/data/partnerPlatforms.ts`
- `src/features/resources/ResourcesPage.tsx`
- `src/data/crossSubjectBlocks.ts`

### 6. Rewards exist visually, but fulfillment logic is incomplete

- Redeeming a reward only deducts points locally.
- Schema already includes `rewards_redemptions`, but UI does not create redemption records or approval state.

Impact:

- Reward history, approvals, and parent fulfillment are missing.
- Gamification cannot be audited or safely expanded.

Primary files:

- `src/features/rewards/RewardsPage.tsx`
- `pb_migrations/schema.json`

### 7. AI tutor is not safe to switch to hosted production use

- Default provider is mock, which is fine for demo.
- If switched to OpenAI, the browser sends a real bearer token directly from the client.

Impact:

- API key exposure risk.
- No server-side moderation, logging, rate limiting, or guardian safety controls.

Primary files:

- `src/services/llmService.ts`

### 8. There are two overlapping app architectures in the repo

- The active entrypoint loads `src/app/App.tsx`.
- An older app shell still exists in `src/App.tsx` with its own auth context under `src/contexts/AuthContext.tsx`.
- Both trees are linted, which increases maintenance noise and makes it easier to fix the wrong layer.

Impact:

- Higher risk of regressions and duplicated logic.
- Slower onboarding and harder production hardening.

Primary files:

- `src/main.tsx`
- `src/app/App.tsx`
- `src/App.tsx`
- `src/contexts/AuthContext.tsx`

## Database and Data Coverage

### Current database shape

PocketBase schema includes:

- families
- learners
- schedules
- blocks
- vr_sessions
- french_lessons
- artifacts
- progress
- competencies
- points
- rewards_redemptions
- ai_conversations

### Important data gaps

- No live database was available to inspect seeded data.
- There is no clear seed or fixture path for families, learners, pods, projects, or activity catalogs.
- Curriculum and pod content is only partially wired into the runtime.
- Only one curriculum export is active in `src/data/curriculum/index.ts`.
- `ALL_PODS` excludes additional elective and AI/ML pod libraries even though those datasets exist.
- Yearly curriculum pods are converted into display pods with empty `weeks`, so they are not executable as real schedules.

Implication:

The content strategy is strong, but the persisted runtime data model is not yet rich enough for a live, portfolio-driven product across ages and subjects.

## Backend Logic Improvements

### Must-do before live use

1. Replace local-only auth with real PocketBase auth and family ownership rules.
2. Unify data access around one PocketBase service layer.
3. Decide whether `progress` is aggregate-per-day or per-block, then align schema, types, and UI to that decision.
4. Add create/upsert flows for families, learners, artifacts, and points.
5. Add migration-backed collections for achievements, integration accounts, sync jobs, imported activity, and project submissions if those are intended product features.
6. Add server-side AI proxy instead of browser-direct model calls.
7. Add automated tests around auth, sync, progress aggregation, and reward redemption.

### Recommended domain additions

- `integration_accounts`: learner/family links to external tools, scopes, refresh metadata, status
- `integration_sync_jobs`: last run, source, record counts, error state
- `external_activity_events`: imported lessons, sessions, assignments, minutes, mastery, streaks
- `projects`: project briefs, rubric, materials, artifact targets, difficulty, age band
- `project_steps`: checklists, scaffolds, estimated time, adult support needed
- `time_support_profiles`: transition warnings, visual countdown style, chunk length, energy windows, sensory supports
- `achievement_unlocks`: separate from static achievement catalog
- `reward_templates` and `reward_redemptions`

## UI and Product Improvements

### Current UI opportunities

- Make "what is real vs planned vs suggested" visually explicit.
- Remove or label demo data so families do not confuse examples for saved progress.
- Add stronger task-state clarity: now, next, later, paused, done, needs-help.
- Add learner-facing "I am stuck" and "I need a reset" actions at the block level.
- Add adult-facing recovery workflows after interrupted days.
- Add project views that tie lessons to tangible build outputs and portfolio evidence.

### Neurodivergent-friendly improvements

- Visual time strip with color-coded passage of time, not only clock times.
- Transition countdowns at 10, 5, 2, and 1 minute.
- "First -> Then -> Next" mode for reduced cognitive load.
- Adjustable schedule granularity: tiny chunks, standard, deep-focus.
- Sensory-aware work modes: quiet, movement, low-light, low-text, audio-first.
- Energy-aware schedule reshuffling based on check-in.
- Clear done states with tangible closure and reflection prompts.
- Optional task decomposition when a learner marks a block as overwhelming.
- Recovery plans for missed days without shame language.

## Gamification Direction

The current points/rewards system is a base layer, not yet a full loop.

### Strong additions

- Quest chains: short narrative missions tied to curriculum pods
- Build badges: evidence-backed achievements for real artifacts, not only point totals
- Mastery maps: visible progress across problem solving, creativity, communication, and perseverance
- Daily momentum: celebrate re-entry after a break, not only perfect streaks
- Team quests: sibling/family collaboration bonuses
- Reflection streaks: reward explaining thinking, not just finishing tasks
- "Experiment tokens": unlock weird/fun side quests and invention challenges
- Showcase seasons: monthly exhibitions, demos, or mini fairs

### Best-fit mechanics for your product vision

- Portfolio-first rewards over shallow streaks
- Problem-solving badges tied to strategy use: compare, test, revise, explain, debug
- Real-world build loops after concept learning
- Choice unlocks that let the learner decide how to apply the concept

Examples:

- Linear algebra -> build a recommendation toy, image transform demo, or game movement matrix experiment
- Regression -> predict plant growth, sports scores, or savings goals with a simple model
- Anatomy -> build a hand tendon model with cardboard, string, and straws
- Chemistry -> safe kitchen or home reaction labs with reflection and evidence capture

## External Connector Roadmap

### Phase 1

- Curated links plus manual import
- Let learners attach external screenshots, links, notebooks, or certificates
- Normalize imported work into artifacts and progress events

### Phase 2

- Account linking and sync for tools with viable APIs or export paths
- Track minutes, completions, mastery, streaks, and created work
- Show source confidence and sync freshness

### Phase 3

- Intelligent cross-platform progress model
- Recommend next LumiPods projects based on external activity
- Auto-suggest portfolio entries from imported work

### Likely targets

- Scratch
- Khan Academy
- Duolingo
- Google Colab / notebook links
- GitHub for code artifacts
- Kaggle notebooks or dataset challenge completions

## Suggested Priority Order

1. Fix auth and family persistence.
2. Unify schema and service layer.
3. Remove or isolate mock data from live views.
4. Define the real progress model.
5. Ship reward redemption persistence and approvals.
6. Add manual external activity capture.
7. Expand executable curriculum and project content.
8. Add neurodivergent support model and UI controls.

## Product Direction

The strongest differentiator here is not "AI tutor" or "homeschool dashboard". It is:

"A confidence-building, project-based learning operating system that turns concepts into real-world builds, visible evidence, strategic thinking habits, and pride."

That is a compelling product direction. The codebase already contains pieces of that vision, but the live connector layer, production auth, executable curriculum depth, and evidence-backed progression model still need real implementation before launch.
